# 設計判断事項

Nablarch クラス可視化ツールの実装前に判断が必要な事項をまとめます。

---

## Q-VIZ-01: 初期俯瞰の簡略化方針

2,681ノードを1枚のキャンバスに描画する際、初期ビューの表示方式を選んでください。

- **Option A: パッケージグループ化** — `nablarch.fw.web.*` などパッケージ単位でノードをグループ表示。ズームインでクラス単位に展開。（視認性高・実装コスト中）
- **Option B: アーティファクトグループ化** — `nablarch-fw-web`（JAR単位）でグループ表示。（境界が明確・実装コスト低）
- **Option C: LOD（Level of Detail）** — ズームレベルに応じて自動切替。全体表示時は点のみ、ズームインでラベル表示。（シームレス・実装コスト高）
- **Option D: フィルタ前提（絞り込み必須）** — 初期表示はメインアーティファクト（`nablarch-fw-web`, `nablarch-core`, `nablarch-fw-batch` 等10件程度）のみ。全表示はフィルタで選択。

### 決定 (2026-05-20)

**採用: Option C（LOD — Level of Detail）**

理由: 6,000超ノードの俯瞰時に全ラベル描画は重過ぎるため、ズームレベルに応じた表示切替が必要。

## Q-VIZ-02: デプロイ形態の希望

設計では3つのデプロイ方式を用意しました（詳細: [docs/architecture.md](docs/architecture.md)）。

- **Option A: 静的ファイル配布** — ローカルCLIで解析 → JSON生成 → `npx serve .`で閲覧。GitHub Pages対応。**PoC推奨**。
- **Option B: バックエンドサーバー常駐** — UIから「このバージョンを解析」ボタンで解析起動。Node.js常駐プロセスが必要。
- **Option C: Dockerコンテナ** — `docker compose up`で全部入り起動。チーム展開時向け。

PoC（Phase 1）はOption Aで開始し、必要に応じてBまたはCに移行する計画ですが、**当初からバックエンドUI（解析ボタン）が必要かどうか**をご確認ください。

### 決定 (2026-05-20)

**採用: Option A（静的ファイル配布）**

理由: PoC段階ではサーバー管理コスト不要。`npx serve` で十分。

## Q-VIZ-03: 描画優先関係性

初期表示（デフォルトフィルタ）でどの関係性を表示するか選んでください。

- **案A（推奨）: 継承+実装のみ（EXTENDS + IMPLEMENTS）** — エッジ数が少なく見やすい。アーキテクチャ構造が明確。
- **案B: 依存関係も含む（EXTENDS + IMPLEMENTS + DEPENDS）** — コンポーネント間の結合が見える。エッジ数は3〜4倍になる見込み。
- **案C: 全関係性（EXTENDS + IMPLEMENTS + USES + CONTAINS + DEPENDS）** — 完全な依存図。初期表示はヘアボール状態になる可能性大。

### 決定 (2026-05-20)

**採用: 案A（継承+実装のみ・EXTENDS + IMPLEMENTS）**

理由: まず最重要の関係性に絞り、ノイズを排除。

## Q-VIZ-05: その他の設計判断事項

以下の点についてご意見があればお知らせください（特段のご指示がなければ設計書の推奨案で進めます）。

- **ClassGraph vs ASM**: 推奨はClassGraph（学習コスト低・JAR一括スキャン対応）。ASMは高速だが実装コスト高。
- **UIフレームワーク**: PoC（Phase 1–3）はVanilla JSで実装予定。React化はPhase 4以降。
- **テストクラスの含む/含まない**: デフォルトはmainクラスのみ（2,681ノード）。テストクラスを含める場合は+3,391ノード（計6,000超）になりパフォーマンスリスクが高まります。

### 決定 (2026-05-20)

- **解析ライブラリ**: ASM（ClassGraphではなくASM採用。実装コスト高であることは承知の上）
- **UIフレームワーク**: React（Phase 1から採用。Vanilla JSではない）
- **テストクラス**: 含む（計6,000超ノード。性能リスクは承知の上で採用）
