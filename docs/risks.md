## R01 — 初期描画パフォーマンス（2,681ノード）

**リスク**: Cytoscape.js で2,681ノードおよび8,000件以上のエッジを描画すると10秒を超え、ユーザーエクスペリエンスが低下する可能性があります。

**発生確率**: 高  
**影響度**: 高 — 対処しないとツールが使い物にならない

**背景**: Cytoscape.js のドキュメントでは、特にエッジのグラフサイズ増加に伴いパフォーマンスが低下することが明示されています。2,681件というノード数は技術選定調査の当初見積もり（500〜2,000件）を約30%上回っています。

**対策**:

1. **事前計算済みレイアウト（必須）**: 解析時に fcose レイアウトを1回実行し、`x`/`y` 座標を `classes.json` に保存。以降の読み込み時は `layout: 'preset'` を使用して高コストな力指向計算を完全にスキップ。プリセットレイアウトでの一般的な描画時間: 1〜3秒。

2. **Canvas レンダラオプション**:
   ```js
   cytoscape({
     textureOnViewport: true,   // パン/ズーム中に Canvas スナップショット — 再描画を防ぐ
     hideEdgesOnViewport: true, // インタラクション中にエッジを非表示 — 大幅なパフォーマンス改善
     motionBlur: true,          // ジャンクの知覚を軽減
     pixelRatio: 1              // DPI スケーリングのオーバーヘッドを回避
   });
   ```

3. **初期ビュー用 LOD（詳細度制御）**: ズーム < 0.3 の場合、個別クラスノードの代わりにパッケージグループ複合ノードを描画。92リポジトリは概ね50〜70のユニークなパッケージプレフィックスに対応し、俯瞰に適した数量です。

4. **エッジ遅延読み込み**: 初期ノードのみの描画完了後に `relations.json` を読み込む。ユーザーには即座にノードが表示され、エッジは1〜2秒後に現れます。

5. **WebGL フォールバック**: 全ての最適化後でも Cytoscape.js の Canvas パフォーマンスが不十分な場合、Sigma.js（WebGL レンダラ）に切り替える。JSON スキーマはフロントエンドに依存しない設計のため、データ層は変更不要です。

**Phase 1 ゲート**: 初期実装後に実際の描画時間を計測する。ステップ1〜4適用後も10秒を超える場合は、Phase 2 前に Sigma.js へのエスカレーションを行う。

## R02 — バージョン間のレイアウト一貫性

**リスク**: バージョン A からバージョン B に切り替えた際、レイアウト座標が大幅に変化し（新クラスの追加や削除のため）、2バージョン間の視覚的な比較が困難になる可能性があります。

**発生確率**: 中（Nablarch の変更は漸進的ですが、クラスの追加は発生します）  
**影響度**: 中 — バージョン比較体験が低下する

**対策**:

1. **安定したレイアウトのアンカリング**: 新バージョン解析時、両バージョンに存在するクラスについては前バージョンのレイアウト座標を再利用する。新規追加クラスのみ新たな fcose 座標を付与する。

2. **差分モード（Phase 3以降）**: 選択した2バージョン間で新規・削除・変更されたノード/エッジを色分けする。これによりレイアウト変更が混乱ではなく意味のある情報となる。

3. **バージョンごとの独立レイアウト**: 各バージョンは自身の `x`/`y` を `classes.json` に保存する。バージョン切り替え時はそのバージョンの事前計算済みレイアウトを読み込み、常に安定した状態を保つ。

## R03 — 解析パイプラインの失敗

**リスク**: ネットワークタイムアウト（GitHub API レート制限、クローン低速）、Maven Central の利用不可、ClassGraph スキャンエラーなどにより、解析パイプラインが途中で失敗する可能性があります。

**発生確率**: 中（GitHub API 制限あり; 92リポジトリ × シャロークローンは非軽量）  
**影響度**: 中 — 解析を再実行する必要があり、部分的な出力は破棄される

**対策**:

1. **インクリメンタルクローン**: 前回の実行からの既存シャロークローンを再利用する。バージョンタグが異なる場合のみ再クローンする。

2. **`meta.json` による進捗追跡**: 各パイプラインステップでステータスを書き込む（`queued → cloning → analyzing → generating → done`）。再実行時は出力が既に存在するステップをスキップする。

3. **GitHub API レート制限**: `gh auth login` で認証; 認証済みリクエストのレート制限は毎時5,000件（未認証は60件）。

4. **タイムアウト処理**: 解析ジョブに15分の全体タイムアウトを設定する。超過した場合はエラーメッセージ付きで `status: "failed"` に設定する。

5. **部分リトライ**: 特定のリポジトリのクローンが失敗した場合（アーカイブ済みや空リポジトリ等）、失敗を記録して残りのリポジトリを処理し続ける。スキップしたリポジトリは `meta.json` に記録する。

## R04 — `pom.properties` 不在 → GAV 抽出失敗

**リスク**: Nablarch JAR に `META-INF/maven/*/pom.properties` が含まれていない場合、ClassGraph が `groupId`/`artifactId`/`version` を抽出できない可能性があります。

**発生確率**: 低（Maven 標準 JAR にはこのファイルがデフォルトで含まれる）  
**影響度**: 中 — アーティファクト単位の色分けが失敗し、全ノードが同色で表示される

**対策**:

1. **ファイル名フォールバック**: JAR ファイル名を解析する。`nablarch-fw-web-6.0.0.jar` → `artifactId=nablarch-fw-web`, `version=6.0.0`。

2. **手動オーバーライドファイル**: 例外ケースのため、JAR ファイル名を GAV 座標にマッピングする `tools/analyzer/artifact-overrides.json` を管理する。

3. **検証ステップ**: Phase 1 実装中に全 JAR ファイルを列挙し、`pom.properties` の存在を確認してからアプローチを確定する。

## R05 — ヘアボールグラフ: 全エッジの同時表示

**リスク**: 2,681ノードと最大8,000〜15,000件のエッジを同時に表示すると、個別の関係性を区別できない「ヘアボール」グラフになり、デフォルト状態でツールが視覚的に使い物にならなくなる可能性があります。

**発生確率**: 高（クラス階層可視化では広く知られた問題）  
**影響度**: 高 — ツールがデフォルト状態で視覚的に機能しなくなる

**対策**:

1. **デフォルトフィルタ: EXTENDS + IMPLEMENTS のみ**: 初期ビューでは継承と実装関係のみを表示する。USES・CONTAINS・DEPENDS はオプトイン（チェックボックスがデフォルトでオフ）。これにより初期エッジ数を推定500〜2,000件に削減できます。

2. **N段階展開**: 非俯瞰モードでは、フォーカスクラスの展開された近傍に接続されたエッジのみを表示し、残りを非表示にする。

3. **最小ズームでのエッジ非表示**: ズーム < 0.15（全体俯瞰）では全エッジを非表示にする。俯瞰はノード分布のみを表示し、ズームインすればエッジが表示される。

4. **エッジ数ガード**: フィルタ変更で可視エッジが5,000件を超える場合、「N件のエッジが表示されます。パフォーマンスが低下する可能性があります。続行しますか？」という確認ダイアログを表示する。

## R06 — エッジ数爆発（USES 関係性）

**リスク**: `USES` 関係性（インポート参照・フィールド参照・メソッド呼び出し）は数万件のエッジを生成し、インタラクティブな描画に適した範囲を大幅に超える可能性があります。

**発生確率**: 高（1つのクラスが20件以上の他クラスをインポートすることがある）  
**影響度**: 高 — JSON ファイルサイズの爆発; 描画のフリーズ

**対策**:

1. **選択的抽出**: Phase 3 では `USES` に対して **フィールド型依存のみ** を抽出する（メソッド呼び出し依存は除外）。これにより最も構造的に重要な関係性のみを残せます。

2. **クラス単位の遅延読み込み**: 全 `USES` エッジをグローバルに読み込む代わりに、クラスがフォーカスノードとして選択された時のみ読み込む（REST API またはオンデマンド JSON チャンク）。

3. **エッジ数上限**: `relations.json` のエッジ数を20,000件でハードリミットする。解析中に上限を超えた場合、どの関係性タイプが除外されたかをログに記録する。

4. **フェーズゲーティング**: `USES`/`CONTAINS`/`DEPENDS` の抽出は Phase 3 の作業です。Phase 1 と Phase 2 は管理可能なことが分かっている `EXTENDS`/`IMPLEMENTS` のみで動作します。

## R07 — Safari ブラウザ互換性

**リスク**: Safari（特にバージョン16未満）には Cytoscape.js が使用する Canvas API 機能に既知の制限があり、Safari 15 以前では WebGL2 サポートが限定的でした。

**発生確率**: 低（主要ユーザーはモダンブラウザを使用）  
**影響度**: 低 — Safari ユーザーのみに影響

**対策**: まず Chrome と Firefox でテストする。Safari はベストエフォート。Chrome/Firefox がサポートブラウザであることを README に明示する。

---

## R08 — Cytoscape.js のパフォーマンス上限

**リスク**: 全ての Canvas 最適化を適用しても、スペックの低いハードウェアでは Cytoscape.js が2,681ノードで10秒の描画目標を達成できない可能性があります。

**発生確率**: 中  
**影響度**: 高 — Sigma.js への切り替えが必要になる場合がある（3〜5日の作業）

**対策**:

1. **Phase 1 でのベンチマーク**: 基準マシン（目標: モダンなノートPC、RAM 8GB、GPU なし）で実際のフレーム時間を計測する。

2. **フォールバック計画の準備**: データスキーマはフロントエンドに依存しません。Cytoscape.js がベンチマークを失敗した場合、描画レイヤーを Sigma.js に移行する。JSON 読み込み・フィルタロジック・バージョン管理コードは変更なし。

3. **成功基準**: 2,681ノードで描画時間 < 10秒 かつ パン/ズーム > 30FPS。どちらかが失敗した場合は Sigma.js へエスカレーションする。

## R09 — 将来の Nablarch バージョンとの ClassGraph 非互換

**リスク**: 将来の Nablarch バージョンが Java の新機能（シールクラス・複雑なジェネリクスを持つレコード等）を使用し、ClassGraph が正しく解析できずに関係性が誤ったまたは不完全な出力になる可能性があります。

**発生確率**: 低（ClassGraph は積極的にメンテナンスされ Java リリースに追随）  
**影響度**: 中 — 解析出力がサイレントに不完全になる

**対策**:

1. **ClassGraph バージョン固定**: `pom.xml` に固定バージョンを使用する（現在: `4.8.184`）。互換性を確認した後のみ更新する。

2. **クラス数検証**: 解析後、`meta.json.total_classes` を count-classes ツールの `class-count.csv` と比較する。10%を超える乖離があれば警告を発する。

3. **変更ログ監視**: ClassGraph の GitHub リリースを購読し、破壊的変更を早期に把握する。
