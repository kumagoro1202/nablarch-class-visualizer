# Nablarch クラス可視化ツール — システムアーキテクチャ設計書

**バージョン**: 1.0  
**作成日**: 2026-05-20  
**技術基盤**: 技術選定調査 — 第1位: Cytoscape.js × ClassGraph

---

## 1. アーキテクチャ概要

システムは明確なコントラクトで接続された2つの独立したコンポーネントに分割されます:

```
┌─────────────────────────────────────────────────────────────┐
│  解析パイプライン（バックエンド / CLI）                        │
│                                                             │
│  入力:  Nablarch バージョンタグ（例: v5.4.0）                  │
│  処理:  リポジトリクローン → ClassGraph スキャン → JSON生成   │
│  出力:  data/versions/{version}/ (4つの JSON ファイル)        │
└───────────────────────┬─────────────────────────────────────┘
                        │ JSON ファイル（ローカルファイルシステム）
┌───────────────────────▼─────────────────────────────────────┐
│  描画フロントエンド（ブラウザ）                               │
│                                                             │
│  入力:  data/versions/{version}/*.json                      │
│  処理:  JSON 読込 → Cytoscape.js 描画 → 探索 UI             │
│  出力:  ブラウザ上のインタラクティブグラフ                    │
└─────────────────────────────────────────────────────────────┘
```

**設計の核心原則**: 解析パイプラインと描画フロントエンドは実行時依存を持ちません。パイプラインは一回限りのバッチジョブであり、フロントエンドは事前計算済みデータを読み込む静的Webアプリケーションです。

---

## 2. 解析パイプライン

### 2.1 入力

- **Nablarch バージョン**: `nablarch` GitHub Organization のリリースタグ（例: `v5.4.0`）またはコミット SHA。
- **リポジトリリスト**: `gh repo list nablarch --limit 200` で実行時に取得（または `data/repo-list.json` にキャッシュ済みのもの）。

### 2.2 処理ステップ

```
ステップ 1: リポジトリリストの解決
        gh repo list nablarch → src/main/java を持つリポジトリを抽出

ステップ 2: 対象バージョンでリポジトリをクローン / 更新
        git clone --depth 1 --branch {version} {repo_url} {work_dir}/{repo}
        （シャロークローン; バージョンが一致する場合は既存クローンを再利用）

ステップ 3: JAR アーティファクトの収集
        mvn dependency:copy-dependencies -DoutputDirectory={work_dir}/jars
        （または: pom.xml の座標から Maven Central 経由でダウンロード）

ステップ 4: ClassGraph スキャン
        ClassGraph().overrideClasspath(jarList)
                    .enableAllInfo()
                    .scan()
        → 抽出: ClassInfo リスト（スーパークラス・インタフェース・フィールド・メソッドを含む）

ステップ 5: GAV 解決
        各 JAR に対して META-INF/maven/*/pom.properties を読み取る
        → groupId, artifactId, version
        フォールバック: JAR ファイル名を解析（例: nablarch-fw-web-6.0.jar）

ステップ 6: JSON 生成
        → classes.json   （ノードデータ）
        → relations.json （エッジデータ）
        → artifacts.json （アーティファクトメタデータ + 色割り当て）
        → meta.json      （解析メタデータ）

ステップ 7: インデックス更新
        → data/versions/index.json （新バージョンエントリを追記）
```

### 2.3 実装オプション

パイプラインは以下のいずれかで実装できます:

| オプション | 技術 | 備考 |
|---------|------|------|
| CLI スクリプト | Java（Maven プラグイン）または Bash + `mvn` | 最もシンプル; オンデマンドでローカル実行 |
| REST API エンドポイント | Node.js または Python（Flask/FastAPI） | オプション B/C デプロイに必要 |
| GitHub Actions ワークフロー | YAML 定義の CI ジョブ | 新リリースの定期解析に有用 |

PoC フェーズでは **CLI スクリプト**で十分です。REST API ラッパーは Phase 2 で追加します。

### 2.4 処理時間の見積もり

92リポジトリ・2,681型宣言に基づく試算:

| ステップ | 推定所要時間 |
|---------|------------|
| リポジトリクローン（92件 × シャロー） | 5〜10分（ネットワーク依存） |
| ClassGraph スキャン（全 JAR） | 30〜60秒 |
| JSON 生成 | 10秒未満 |
| **合計** | **約10〜15分** |

2回目以降の実行は既存クローンを再利用し、変更のないリポジトリをスキップするため、増分更新では2分未満に短縮されます。

## 3. 描画フロントエンド

### 3.1 技術スタック

| コンポーネント | 技術 | 選定理由 |
|------------|------|---------|
| グラフ描画 | **Cytoscape.js**（MIT） | 全 REQ をカバー、学習コスト2〜3日、豊富なプラグインエコシステム |
| レイアウトエンジン | **cytoscape-fcose** プラグイン | 1,000〜3,000ノードに対応; 複合ノードをサポートする力指向レイアウト |
| 検索 / ハイライト | **cytoscape-view-utilities** プラグイン | `highlightNeighbors()` と `zoomToSelected()` が標準搭載 |
| UI フレームワーク | **Vanilla JS**（Phase 1〜3）/ React（Phase 4以降） | PoC の依存関係を最小化; チーム協業が必要になった場合のみ React に移行 |
| ビルドツール | **Vite** | 高速 HMR、ES モジュール、最小限の設定 |
| データ読み込み | `fetch()` + JSON | 静的ファイル; フロントエンドのみのデプロイではバックエンド不要 |

### 3.2 フロントエンドコンポーネント構成

```
src/
├── index.html           # シングルページアプリのシェル
├── main.js              # エントリポイント: データ読込、Cytoscape 初期化
├── graph/
│   ├── init.js          # Cytoscape インスタンスのセットアップ、レイアウト設定
│   ├── style.js         # ノード/エッジのスタイルルール（アーティファクト色分け、エッジタイプ）
│   ├── layout.js        # レイアウト選択と事前計算済み座標フォールバック
│   └── events.js        # クリック・ホバー・ズームイベントハンドラ
├── ui/
│   ├── search.js        # デバウンス + ハイライト付き検索ボックス
│   ├── filters.js       # 関係性タイプおよびアーティファクトフィルタコントロール
│   ├── expand.js        # N段階展開コントロール（+1/−1/全展開/リセット）
│   ├── version.js       # バージョンセレクタと解析起動
│   └── legend.js        # アーティファクト色レジェンド
└── data/
    └── loader.js        # data/versions/{version}/ から JSON を読み込み検証
```

### 3.3 2,681ノードのパフォーマンス戦略

| 手法 | 詳細 |
|-----|------|
| **事前計算済みレイアウト** | fcose を1回実行し、ノード座標を `classes.json` に保存。読み込み時は `layout: 'preset'` を使用し、高コストな計算をスキップ。 |
| **ビューポートテクスチャ** | パン/ズーム中に Cytoscape オプションの `textureOnViewport: true` を有効化。 |
| **エッジの非表示** | インタラクション中に `hideEdgesOnViewport: true` を有効化; アイドル時に復元。 |
| **詳細度制御（LOD）** | ズームが閾値以下の場合、個別クラスノードの代わりにパッケージグループ複合ノードを描画。 |
| **遅延リレーション読み込み** | 初期描画後に `relations.json` を読み込む。最初のビューはノードのみ表示; エッジはバックグラウンドで読み込み。 |

### 3.4 バージョン管理 UI

```
┌────────────────────────────────────────────────────────┐
│ バージョン: [v5.4.0 ▼]  [新バージョンを解析]            │
│ 解析日: 2026-05-19  クラス数: 2,681                     │
└────────────────────────────────────────────────────────┘
```

- バージョンセレクタは `data/versions/index.json` から生成される。
- 「新バージョンを解析」ボタンでモーダルを表示: バージョンタグの入力ボックス → 解析 API を呼び出す（オプション B/C）またはCLIコマンドを表示（オプション A）。
- 進捗は `data/versions/{version}/meta.json` の `status` フィールドをポーリングして取得。

## 4. デプロイオプション

### オプション A: 静的ファイル配布（PoC および個人利用向け推奨）

```
nablarch-class-visualizer/
├── index.html         （フロントエンドエントリポイント）
├── assets/            （Vite ビルド出力: JS, CSS）
└── data/
    └── versions/
        ├── index.json
        └── v5.4.0/
            ├── classes.json
            ├── relations.json
            ├── artifacts.json
            └── meta.json
```

- 解析はサーブ前に **ローカルで** CLI コマンドとして実行する。
- フロントエンドは `npx serve .` や任意の静的ファイルサーバー（Python `http.server`、Nginx 等）で配信する。
- GitHub Pages デプロイ: `data/` とビルド済みアセットをリポジトリに push; 解析出力をリポジトリにコミット。
- **実行時にサーバープロセス不要。**
- 解析起動 UI は「ローカルで解析を実行する」手順の案内に置き換える。

**メリット**: インフラコストゼロ、初回データ生成後は完全オフライン動作、共有が容易（zip して送るだけ）。  
**デメリット**: 解析は手動実行が必要; バージョン更新にはローカルCLI実行と再デプロイが必要。

---

### オプション B: バックエンドサーバー

```
┌──────────────┐        REST API         ┌────────────────────┐
│   ブラウザ    │ ←───────────────────── │  バックエンドサーバー │
│  (Vite SPA)  │                        │  Node.js / Python  │
└──────────────┘                        │  ├── /api/versions  │
                                        │  ├── /api/analyze   │
                                        │  └── /api/status    │
                                        └─────────┬──────────┘
                                                  │ 起動
                                        ┌─────────▼──────────┐
                                        │  解析 CLI           │
                                        │  (Java / Bash)     │
                                        └────────────────────┘
```

- 解析は UI から起動: POST `/api/analyze?version=v5.4.0` → サーバーが CLI プロセスを起動。
- 進捗ポーリング: GET `/api/status/{version}` → 現在のステップとログ末尾を返す。
- JSON データは同一サーバーから配信: GET `/data/versions/v5.4.0/classes.json`。
- **常駐サーバープロセスが必要。**

**メリット**: UI 主導のフルワークフロー; ローカル CLI 不要。  
**デメリット**: サーバーセットアップが必要（Node.js/Python 環境、プロセス管理）。

---

### オプション C: Docker コンテナ

```
docker run -p 8080:8080 -v ./data:/app/data nablarch-viz:latest
```

- Docker イメージにバックエンドサーバー（オプション B）とフロントエンドをバンドル。
- `data/` はコンテナ再起動をまたいで永続化するためボリュームとしてマウント。
- Docker Compose バリアントには解析ワークスペース用のボリュームを含める。

**メリット**: ワンコマンド起動; 再現性のある環境; チームへの配布が容易。  
**デメリット**: ホストマシンに Docker が必要; イメージビルドのセットアップ時間が発生。

---

### 推奨まとめ

| シナリオ | 推奨オプション |
|---------|-------------|
| 個人 PoC（Phase 1〜2） | **オプション A** — インフラゼロ、最速で開始可能 |
| 個人開発、UIドリブンの解析を希望 | **オプション B** — ローカルホスト上の軽量サーバー |
| チーム共有 / 環境統一 | **オプション C** — データボリュームマウント付き Docker Compose |

**PoC（Phase 1）ではオプション A を推奨します。** アーキテクチャはフロントエンドやデータスキーマを変更せずに後からオプション B/C に移行できるよう設計されています。

## 5. データフローのまとめ

```
[Nablarch GitHub リポジトリ]
        ↓ git clone（シャロー、バージョンごと）
[ローカル作業ディレクトリ: /tmp/nablarch-viz-work/]
        ↓ Maven ビルド → JAR 収集
[JAR ファイル: nablarch-fw-web-6.0.jar, nablarch-core-6.0.jar, ...]
        ↓ ClassGraph スキャン + pom.properties による GAV 抽出
[Java オブジェクト: ClassInfo（スーパークラス・インタフェース・フィールド付き）]
        ↓ JSON シリアライズ（カスタムアダプタ）
[data/versions/v5.4.0/classes.json + relations.json + artifacts.json + meta.json]
        ↓ ブラウザの fetch()
[Cytoscape.js グラフ: ノード + エッジ]
        ↓ ユーザー操作
[フィルタ済みサブグラフ: N段階展開・関係性タイプフィルタ・アーティファクトフィルタ]
```
