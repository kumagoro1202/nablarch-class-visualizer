**バージョン**: 1.0  
**作成日**: 2026-05-20

## 概要

実装は4フェーズに分割されます。各フェーズで動作・デモ可能な成果物を生み出します。Phase 1〜3は必須、Phase 4はチーム協業の必要性に応じてオプションです。

| フェーズ | 名称 | 想定期間 | デプロイ方式 |
|--------|------|---------|------------|
| 1 | PoC — 単一バージョン・コア可視化 | 1〜2週間 | オプション A（静的） |
| 2 | バージョン管理 | 1〜2週間 | オプション A または B |
| 3 | フル探索 UI | 2〜3週間 | オプション A または B |
| 4 | チーム配布 | オプション | オプション C（Docker） |

## Phase 1: PoC — 単一バージョン・コア可視化

**目標**: Cytoscape.js を使用して2,681件の Nablarch ノードをブラウザ上でインタラクティブに描画できることを証明し、アーティファクト色分けと基本的な検索機能を実装する。

### スコープ

- 解析パイプライン: 92件の Nablarch リポジトリを ClassGraph でスキャン; `EXTENDS` と `IMPLEMENTS` 関係のみを抽出。
- フロントエンド: 単一ページの Cytoscape.js アプリ; 事前生成した JSON を読み込んでノードとエッジを描画。
- レイアウト: fcose レイアウト（1回実行後、座標を `classes.json` に保存）。
- 機能:
  - 初期ロードで全ノードを表示（俯瞰モード）
  - レジェンド付きアーティファクト色分け
  - クラス名検索 → ハイライト → 自動中央表示
  - マウスホイールズームとキャンバスパン
  - ノードクリック → クラス詳細パネルを表示（FQCN・アーティファクト・タイプ・モディファイア）

### Phase 1 スコープ外

- N段階展開コントロール
- 関係性タイプフィルタ
- バージョン管理 UI
- UI からの解析起動

### 成果物

- `tools/analyzer/` — ClassGraph を使用した Java CLI（Maven プロジェクト）
- `data/versions/v{initial}/` — 1つの Nablarch バージョンの JSON 出力
- `src/` — Cytoscape.js フロントエンド（Vanilla JS + Vite）
- `index.html` — 静的配信のエントリポイント

### 工数見積もり（1名）

| タスク | 日数 |
|--------|------|
| Java ClassGraph セットアップ、JAR ダウンロード、GAV 抽出 | 2 |
| JSON シリアライズ（classes + relations + artifacts + meta） | 1 |
| Cytoscape.js セットアップ + fcose レイアウト + スタイルルール | 2 |
| 検索ボックス + ハイライト + 自動中央表示 | 1 |
| アーティファクトレジェンド + カラーパレット | 1 |
| パフォーマンスチューニング（テクスチャ・エッジ非表示・LOD） | 1 |
| **合計** | **約8日** |

### 検証基準

- [ ] ブラウザ（Chrome/Firefox）でエラーなく読み込める
- [ ] 初期描画後、全2,681ノードが表示される
- [ ] 初期描画が10秒以内に完了する
- [ ] 各アーティファクトが異なる色で描画され、レジェンドが正確である
- [ ] "HttpRequestHandler" で検索すると該当ノードがハイライトされカメラが中央に移動する
- [ ] ズームイン/アウトとパンがスムーズに動作する

## Phase 2: バージョン管理

**目標**: 複数の解析済み Nablarch バージョンを選択し、UI から新バージョンの解析を起動できるようにする。

### スコープ

- 解析パイプライン拡張: バージョンタグを CLI 引数として受け付ける; 各ステップで `meta.json` の `status` フィールドを更新する。
- フロントエンド: バージョンセレクタドロップダウン; 進捗表示付き「新バージョンを解析」ボタン。
- デプロイ: オプション A（静的 + ローカル CLI）またはオプション B（軽量 REST サーバーを追加）。
- 各解析完了後に `data/versions/index.json` を自動更新。

### 機能

- `data/versions/index.json` から動的生成されるバージョンドロップダウン
- バージョン切り替え: ページリフレッシュなしに JSON を再読み込み; ズームレベルと有効フィルタを可能な範囲で保持
- 解析起動（オプション A）: モーダルに CLI コマンドを表示
- 解析起動（オプション B）: POST `/api/analyze` → 5秒ごとに `/api/status/{version}` をポーリング; 進捗ステップと推定時間を表示
- 解析失敗時: UI にエラー状態を表示; 末尾20行のログを表示

### Phase 2 スコープ外

- N段階展開
- `USES`・`CONTAINS`・`DEPENDS` 関係性タイプ（引き続き `EXTENDS`/`IMPLEMENTS` のみ）

### 成果物

- `tools/analyzer/` を更新し、`meta.json` への進捗書き込みに対応
- `src/ui/version.js` — バージョンセレクタ + 解析起動コンポーネント
- （オプション B のみ）`server/index.js` — Node.js REST API ラッパー

### 工数見積もり

| タスク | 日数 |
|--------|------|
| `meta.json` に進捗を書き込むパイプライン更新 | 1 |
| `data/versions/index.json` の管理 | 0.5 |
| バージョンセレクタ UI | 1 |
| バージョン切り替え（JSON 再読み込み・状態保持） | 1 |
| 解析起動 + 進捗表示（オプション A: CLI モーダル） | 1 |
| （オプション B）Node.js サーバー + 進捗ポーリング | 2 |
| **合計（オプション A）** | **約4.5日** |
| **合計（オプション B）** | **約6.5日** |

### 検証基準

- [ ] 解析済みバージョンが2件存在し、切り替えるとグラフが更新される
- [ ] バージョンセレクタに各バージョンの解析日とクラス数が表示される
- [ ] 解析起動（オプション A）: 表示される CLI コマンドが正しい
- [ ] （オプション B）解析起動でUIに進捗ステップが表示され、完了後にバージョンセレクタに追加される

## Phase 3: フル探索 UI

**目標**: `requirements.md` で定義された完全な N段階展開・4種類の関係性タイプフィルタ・アーティファクトフィルタ・パッケージ階層フィルタを実装する。

### スコープ

- 解析パイプライン: `USES`・`CONTAINS`・`DEPENDS` 関係性の抽出を追加（ClassGraph フィールド/メソッド解析）。
- フロントエンド: フル探索コントロール; 初期ノード描画後に `relations.json` を遅延読み込み。

### 機能

- **N段階展開**: フォーカスクラス選択; +1/−1/全展開/リセットコントロール
- **関係性タイプフィルタ**: EXTENDS/IMPLEMENTS/USES/CONTAINS/DEPENDS のチェックボックスグループ; リアルタイムエッジ更新
- **アーティファクトフィルタ**: アーティファクトのマルチセレクト; ノードの表示/非表示とカウント表示
- **パッケージ階層フィルタ**: 部分一致テキスト入力; `nablarch.fw.web` 等のパッケージプレフィックスでフィルタ
- **俯瞰最適化**: 低ズーム時にノードをパッケージグループ複合ノードに折り畳む（LOD）
- **エッジ数上限ガード**: 有効フィルタで5,000エッジを超える場合、描画前に警告を表示

### 技術上の注意点

- `relations.json` はサイズが大きくなる可能性があります（全タイプで推定8,000〜15,000エッジ）。遅延読み込みを使用します: ユーザーが最初にフィルタトグルを操作したときにのみファイルを読み込みます。
- N段階展開: メモリ上に「表示ノード」セットを管理; Cytoscape の `batch()` 呼び出しで一括表示/非表示を行い、レイアウト再計算を回避する。
- パッケージグループ LOD: Cytoscape 複合ノードを使用; パッケージプレフィックスで子ノードをグループ化。`cy.zoom()` 変更イベントに基づいて LOD を切り替える。

### 成果物

- `tools/analyzer/` を更新し、フィールド/メソッド依存関係抽出に対応
- `src/ui/expand.js`、`src/ui/filters.js` — 展開・フィルタコンポーネント
- `docs/data-schema.md` を更新し、USES/CONTAINS/DEPENDS エッジのサンプルを追加
- パフォーマンスベンチマークレポート（全エッジ表示時の 2,681ノードでの FPS 計測）

### 工数見積もり

| タスク | 日数 |
|--------|------|
| ClassGraph フィールド/メソッド抽出（USES/CONTAINS/DEPENDS） | 2 |
| N段階展開ロジック + UI コントロール | 3 |
| 関係性タイプフィルタ（チェックボックス + エッジ更新） | 2 |
| アーティファクトフィルタ + パッケージフィルタ | 2 |
| LOD 複合ノード実装 | 2 |
| relations.json の遅延読み込み | 1 |
| パフォーマンスベンチマーク | 1 |
| **合計** | **約13日** |

### 検証基準

- [ ] クラスを選択して「+1レベル」を押すと直接の隣接ノードのみが表示される
- [ ] 「−1レベル」を押すと最外リングが折り畳まれる
- [ ] 「全展開」でフォーカスクラスから推移的に到達可能な全ノードが表示される
- [ ] 「リセット」で俯瞰モードに戻る
- [ ] 「EXTENDS」フィルタを無効化すると全 EXTENDS エッジがビューから消える
- [ ] `nablarch-core` アーティファクトのみを選択すると他のアーティファクトのノードが非表示になる
- [ ] パッケージフィルタ `nablarch.fw.web` でそのパッケージプレフィックスのノードのみが表示される
- [ ] 低ズーム時にノードがパッケージグループに折り畳まれ、ズームインで展開される

## Phase 4: チーム配布（オプション）

**目標**: チームへの共有またはセルフホスト型デプロイ向けにツールをパッケージ化する。

### スコープ

- 解析パイプライン（Java）+ フロントエンドサーバー（Node.js/Nginx）を含む Docker イメージ
- `data/` とワークスペース用ボリュームマウント付き Docker Compose ファイル
- セットアップ手順の README
- CI/CD: 新 Nablarch リリース解析の定期実行用 GitHub Actions ワークフロー（オプション）

### 機能

- `docker compose up` でフルスタック起動
- バインドマウントにより `data/` がコンテナ再起動をまたいで永続化
- GitHub Actions ワークフロー: Nablarch リリースイベントでトリガー → 解析実行 → JSON をリポジトリにコミット

### 成果物

- `Dockerfile` + `docker-compose.yml`
- `docs/deployment.md` — セットアップおよび運用ガイド
- `.github/workflows/analyze-new-release.yml`（オプション）

### 工数見積もり

| タスク | 日数 |
|--------|------|
| Dockerfile（マルチステージ: Java ビルダー + Node サーバー） | 1.5 |
| Docker Compose + ボリューム設定 | 0.5 |
| デプロイドキュメント | 1 |
| GitHub Actions ワークフロー（オプション） | 1 |
| **合計** | **約4日** |

### 検証基準

- [ ] クリーンな環境で `docker compose up` がエラーなく起動する
- [ ] コンテナ内での UI からのバージョン解析が正常に完了する
- [ ] `docker compose down` + `docker compose up` 後も `data/` の内容が維持される
- [ ] （オプション）GitHub Actions ワークフローが Nablarch リリースでトリガーされ `data/` を更新する

## 依存関係と前提条件

|---------|------------|------|
| Java 17+ | Phase 1（ClassGraph 解析） | ClassGraph 4.8.x に必要 |
